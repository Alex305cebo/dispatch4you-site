# Bugfix Requirements Document

## Introduction

В симуляторе переговоров диспетчер-брокер постоянно создаются диалоги с тупиками. Когда диспетчер выбирает варианты низкого качества (4-6: weak/aggressive/fail) на любом шаге N master пути, система направляет его в промежуточные reject пути, которые начинаются с шага 1 вместо финального outcome. Это приводит к несинхронизации: брокер задает вопрос шага 1 reject пути вместо финального ответа об отказе.

Проблема повторяется при создании каждого нового диалога в файлах scenarios-data-vXX.js, что указывает на системную ошибку в логике создания путей диалога.

## Bug Analysis

### Current Behavior (Defect)

1.1 WHEN диспетчер выбирает вариант 4 (weak) на шаге N master пути THEN система направляет в reject путь "weak", который содержит несколько промежуточных шагов начиная с шага 1

1.2 WHEN диспетчер выбирает вариант 5 (aggressive) на шаге N master пути THEN система направляет в reject путь "warning" или "reject_attitude", который содержит промежуточные шаги вместо финального outcome

1.3 WHEN диспетчер выбирает вариант 6 (fail) на шаге N master пути THEN система направляет в reject путь (например "reject_weak", "reject_experience"), который может содержать промежуточные шаги вместо финального outcome

1.4 WHEN диспетчер находится на шаге N master пути и попадает в reject путь с промежуточными шагами THEN брокер задает вопрос шага 1 reject пути вместо финального ответа

1.5 WHEN создается новый диалог THEN промежуточные reject пути создаются автоматически вместо финальных reject путей с одним outcome элементом

### Expected Behavior (Correct)

2.1 WHEN диспетчер выбирает вариант 4 (weak) на шаге N master пути THEN система SHALL направить СРАЗУ в финальный outcome пути "reject_weak_final", который содержит только один элемент с brokerResponse и outcome

2.2 WHEN диспетчер выбирает вариант 5 (aggressive) на шаге N master пути THEN система SHALL направить СРАЗУ в финальный outcome пути "reject_attitude_final", который содержит только один элемент с brokerResponse и outcome

2.3 WHEN диспетчер выбирает вариант 6 (fail) на шаге N master пути THEN система SHALL направить СРАЗУ в соответствующий финальный outcome путь (reject_nomc_final, reject_timing_final, reject_experience_final и т.д.), который содержит только один элемент с brokerResponse и outcome

2.4 WHEN диспетчер попадает в любой reject_*_final путь THEN брокер SHALL дать финальный ответ об отказе без дополнительных вопросов

2.5 WHEN создается новый диалог THEN все reject пути SHALL быть финальными (reject_*_final) и содержать только один элемент с outcome, без промежуточных шагов

2.6 WHEN диспетчер выбирает вариант 1-3 (excellent/good/normal) на шаге N master пути THEN система SHALL направить на шаг N+1 master пути

### Unchanged Behavior (Regression Prevention)

Эта секция описывает поведение, которое работает ПРАВИЛЬНО и НЕ должно измениться после исправления бага. Это защита от регрессии - гарантия, что исправление не сломает то, что уже работает корректно.

3.1 WHEN диспетчер выбирает варианты 1-3 (excellent/good/normal) на любом шаге master пути THEN система SHALL CONTINUE TO направлять на следующий шаг master пути
   - Пример: На шаге 5 выбрал вариант 2 (good) → переход на шаг 6 master пути
   - Это работает правильно, не трогать!

3.2 WHEN диспетчер проходит весь master путь успешно THEN система SHALL CONTINUE TO показывать финальный success outcome с премиальной ставкой
   - Пример: Прошел все 14 шагов с вариантами 1-3 → получил $2,750 (выше posted $2,500)
   - Это работает правильно, не трогать!

3.3 WHEN диалог содержит финальные reject пути формата "reject_*_final" с одним элементом THEN система SHALL CONTINUE TO корректно показывать финальный отказ брокера
   - Пример: В scenarios-data-v20.js все reject_*_final пути работают правильно
   - Это работает правильно, не трогать!

3.4 WHEN диспетчер находится на последнем шаге master пути и выбирает вариант 6 (fail) "I'll take posted rate" THEN система SHALL CONTINUE TO направлять на следующий шаг master пути
   - Пример: На шаге торга выбрал "I'll take posted rate" → это не ошибка, а принятие условий
   - Это работает правильно, не трогать!

3.5 WHEN диалог использует структуру путей с brokerQuestion, dispatcherPrompt, suggestions, path THEN система SHALL CONTINUE TO корректно обрабатывать навигацию между путями
   - Пример: Структура { brokerQuestion, dispatcherPrompt, suggestions: [{ text, quality, analytics, path }] }
   - Это работает правильно, не трогать!
